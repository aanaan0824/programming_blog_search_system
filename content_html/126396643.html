🐳 新闻应具备哪些属性


  • 新闻应具备哪些属性？这些属性是否可以放在同一张表内？
  • 如果不能放在同一张表内，将其属性拆分、存储在其他关系表内，如此即可分析出 “新闻管理系统” 需要几张数据表了。
  • 新闻表属性如下：
    • 编号：设置为主键（递增），整数类型，不可为负数
    • 标题：标题类型（varchar）
    • 作者：网站的新闻都是 “编辑” 撰写的，这些用户有账号、密码以及角色，所以这个信息需要单独保存为一个数据表。新闻表只需要去引用用户表的 id 即可。
    • 类型：类型应该设置为 "可维护性的"，比如时政新闻、娱乐新闻、民生新闻等，在需要的情况下还可以添加如金融新闻、海外新闻等等。如果说设计字段的时候，把 "类型" 字段写死了，再添加新的 "新闻类型" 的时候还得去修改表结构。在使用 "alter table" 修改表结构是要锁表的，锁表的情况下是既不能读取也不能写入的，直到 alter table 结束之后才可以读写数据表。（在数据非常多的情况下，就会影响数据的运行）这里我们希望能够引用 “新闻类型表” 的 id ，所以也需要为 “新闻类型” 单独创建一个数据表。后续如果需要添加 “新闻类型” ，只需要向 “新闻类型表” 添加一条记录即可，不需要去修改 “新闻表” 的表结构即可实现，也就不会产生 “锁表” 的问题了，维护起来也非常的方便。>
    • 内容：内容这个字段就更具有特殊性了，MySQL 的字符串不太适合保存太长的字符串内容，否则读写速度就会非常的慢。一个正常的新闻可能会有几千字，甚至还会存在内嵌的图片，所以就不能使用 MySQL 存储新闻的正文。这里可以在 "MongoDB"数据库创建一个 “新闻内容表” ； “新闻内容表” 的每一个新闻内容也拥有主键编号，主键编号保存在MySQL新闻表的 “内容字段上” ，如此就构成了一个引用的关系。MongoDB数据库非常适合保存海量低价值的数据，特别是数据内容多的情况下，读取速度要比 MySQL 快上很多。所以 “新闻内容” 需要保存在 MySQL 数据表之外才行，要用更快的数据存储引擎才行。
    • 置顶：可以设置成整数的级别，比如说置顶的级别分为 10 级，级别越大，置顶就越优先。
    • 创建时间：时间戳类型
    • 修改时间：时间戳类型
    • 状态：状态的备选值都是可选的状态，而且是固定的。比如说 “草稿状态”、“审核中状态”、“已审核状态” 等等，使用
  • 用户表属性如下：
    • 编号：设置为主键（递增），整数类型，不可为负数
    • 用户：即用户信息
  • 新闻类型表属性如下：
    • 编号：设置为主键（递增），整数类型，不可为负数
    • 类型：即新闻类型，便于维护。
  • 新闻内容表属性如下：（使用 MongoDB 数据库，主键编号与MySQL构成引用的关联）
    • 编号：设置为主键（递增），整数类型，不可为负数
    • 内容：正常的新闻内容

如果说新闻系统再大一些、庞杂一些，可能会用到更多的数据表，但目前我们所实现的是一个非常简单的功能，四张表基本上就够用了。
